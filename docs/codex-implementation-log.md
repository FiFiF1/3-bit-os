# Codex Implementation Log

This file records the current engineering direction for Codex.

## Core decision

3 Bit OS is a balanced ternary operating environment, but normal laptops are binary machines.

That means the project needs two layers:

1. A small binary host layer that can run on real x86_64 laptop hardware.
2. A balanced ternary VM and kernel that form the actual 3 Bit OS.

The binary host exists only to boot the laptop, talk to real hardware, and run the ternary VM.

## Practical laptop path

Build in this order:

1. Desktop VM runner.
2. Ternary assembler and kernel image format.
3. Ternary kernel and shell.
4. x86_64 UEFI application host.
5. Bootable QEMU image.
6. Bootable USB image.
7. Later, a post-ExitBootServices binary micro-host.

Do not start with full GPU drivers, destructive installers, internet support, or POSIX compatibility.

## Important UEFI distinction

There are two hardware boot modes.

### Firmware-hosted mode

The UEFI app boots as `BOOTX64.EFI` and keeps using firmware services.

Useful for:

- First laptop boot.
- GOP framebuffer output.
- UEFI console input.
- Proving the VM and shell can run on real hardware.

Limit:

- The firmware still owns much of the system.

### Owned-hardware mode

The host gets the memory map, calls `ExitBootServices()`, then keeps running without boot services.

Useful for:

- Becoming closer to a real OS.
- Owning memory and devices.
- Long-term installable system work.

Limit:

- Keyboard, storage, timers, and drivers become much harder.

## Hardware reality

### CPU

The CPU will not execute ternary instructions directly. The binary host CPU runs either:

- a ternary interpreter, first version
- a faster block interpreter, later
- a JIT compiler, much later

### GPU

Start with framebuffer graphics only.

First GPU goals:

- clear screen
- draw pixel
- draw rectangle
- draw text
- blit a buffer

Do not attempt NVIDIA, AMD, or Intel hardware acceleration first. Vendor GPU drivers are too large for the first version.

### Storage

Start with file-backed storage inside the desktop runner.

Then add:

- read-only initramfs-like ternary image
- QEMU disk image
- USB image
- later real disk access

## Binary code that will be needed

The project will eventually need binary artifacts:

- UEFI executable: `BOOTX64.EFI`
- packed ternary kernel image
- raw disk image
- qcow2 image for QEMU
- optional ISO image
- optional tiny assembly stubs
- optional future JIT-generated host machine code

Most binary output should be generated by toolchains and image builders. Avoid hand-written machine code unless it is isolated and documented.

## Codex rules

Codex should:

- keep changes small
- implement one layer at a time
- write tests before moving up a layer
- document every binary format
- avoid destructive disk operations
- prefer safe testable runners before real hardware
- keep the ternary kernel separate from the binary host
